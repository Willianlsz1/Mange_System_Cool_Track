feat(pmoc): epic completo — clientes, checklist NBR 13971, PDF formal

Implementa o ciclo PMOC completo (Lei 13.589/2018, NBR 13971, Portaria
GM/MS 3.523/1998) em 6 fases. Posiciona o Pro como tier de conformidade
legal, mantendo o relatório técnico simples disponível pra todos os planos.

Fase 1 — Clientes + dados legais empresa
* Migration 20260425120000: tabela `clientes` (multi-tenant, RLS), FK
  `equipamentos.cliente_id` (ON DELETE SET NULL), 4 colunas em profiles
  (razao_social, cnpj, inscricao_estadual, inscricao_municipal).
* profileModal: nova seção "Dados legais (opcional)" com 4 inputs PMOC.
* PDF cover: caixa "TÉCNICO RESPONSÁVEL" expande pra 42mm quando há
  dados PMOC.

Fase 2 — Tela "Meus clientes" + vínculo
* core/clientes.js: CRUD com Supabase (loadClientes, upsertCliente,
  deleteCliente, validateCnpjOrCpf, formatCnpjOrCpf). Validação CNPJ/CPF
  com dígito verificador, permissiva (warn, não bloqueia).
* state.clientes hidratado on-demand. Não persiste local (vivem no
  Supabase, sem queue offline na fase 2).
* ui/views/clientes.js + ui/components/clienteModal.js: lista com cards,
  busca por nome/CNPJ, empty state, modal de cadastro/edição com 9 campos.
* Equip ↔ cliente: select novo no modal-add-eq (lazy populate), storage
  normalize com fallback `isMissingClienteSchemaError` pra schema antigo.
* Header menu: item "Meus clientes" com data-nav.

Fase 3 — Checklist NBR 13971
* Migration 20260425130000: coluna `checklist JSONB` em registros, check
  constraint jsonb_typeof = 'object', índice GIN parcial.
* domain/pmoc/checklistTemplates.js: 8 templates por tipo (Split Hi-Wall,
  Cassette, Piso-Teto, VRF/VRV, Fan Coil, Chiller, Self/Roof Top, Generic)
  baseados na NBR 13971:2014. Items com id, label, group, mandatory.
* registro.js: novo accordion "Checklist NBR 13971", grid items + 3
  status pills (✓ conforme / ✗ não-conforme / ⊘ N/A) + textarea obs.
* Soft-required: warning amber em preventiva sem checklist (não bloqueia).
* PDF section/checklist.js: tabela formal com agrupamento + status colorido.

Fase 4 — Medições numéricas com unidades
* 20 ids de items distintos marcados `measurable: true` + `unit`
  (psi/kPa/A/V/°C). 42 ocorrências através dos 8 templates.
* registro.js: input numérico inline (inputmode="decimal", aceita vírgula
  pt-BR), unit como suffix visual.
* formatMeasure() helper com formato pt-BR (vírgula decimal, trailing
  zeros removidos, integer sem casa).
* PDF: nova coluna "Medição" entre Status e Obs (alinhada à direita,
  bold, 22mm).

Fase 5 — PDF PMOC formal monocromático
* domain/pdf/pmoc/: módulo dedicado com paleta black/gray/white, fonte
  Times nos títulos institucionais e legal body, helvetica nas tabelas.
* sections/cover.js: capa formal (brand line, título centrado, número
  PMOC YYYY/NN, ano-base, blocos cliente + prestador, base normativa).
* sections/registry.js: cadastro técnico (TAG, equipamento, fluido,
  capacidade, local, periodicidade), filtra por cliente, ordena por TAG.
* sections/schedule.js: matriz 12 meses × equipamentos com R/C/L/P marks
  (realizada/corretiva/limpeza/planejada) coloridas em escala de cinza.
* sections/termo.js: termo de RT formal NBR 13971 com declaração padrão,
  identificação do RT (8 campos), local + data, linha de assinatura.
* pmocReport.js: orquestrador generatePmocPdf() com numeração sequencial
  via localStorage (escopo por user+ano), formato "PMOC YYYY/NN".

Fase 6 — UX final (discovery + dropdown)
* Toolbar: dropdown "Exportar PDF ▾" unifica relatório técnico e PMOC
  formal num único bloco. Click direto = export-pdf (legacy preserved);
  caret abre menu com 2 opções + item "Sobre o PMOC".
* Smart preselect: handler open-pmoc-modal detecta cliente via
  data-cliente-id ou via filtro rel-equip (equip único → cliente do equip).
* Card cliente com ≥3 equipamentos vinculados ganha CTA discreto
  "📜 Gerar PMOC".
* Header menu: itens "Gerar documento PMOC" + "Sobre o PMOC" como
  atalhos globais.
* pmocInfoModal: modal educacional explicando o que é PMOC, base legal,
  comparação lado-a-lado dos 2 PDFs, quando usar cada um.

Pro-gating
* PMOC formal exclusivo do plano Pro (handler faz double-check via
  fetchMyProfileBilling + hasProAccess pra evitar TTL stale do cache).
* Free/Plus veem variante "lock" no modal com CTA "Desbloquear com Pro →".
* Telemetria: pmoc_pdf_generated com {ano, cliente_vinculado,
  equipamentos_count, from_preselect}.

Compatibilidade
* Storage normalize com fallbacks `isMissingClienteSchemaError` e
  `isMissingChecklistSchemaError`: PDFs e cadastros continuam funcionando
  mesmo se as 2 migrations PMOC ainda não rodaram em produção (graceful
  degrade, sem perder dados locais).
